# TIL Repository Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Scaffold a personal TIL repo with MkDocs Material, auto-deploy to GitHub Pages via GitHub Actions, with full-text search and tag browsing.

**Architecture:** Markdown files organised in category folders under `docs/`. MkDocs Material builds the static site locally and the GitHub Actions workflow deploys it to the `gh-pages` branch on every push to `main`. GitHub Pages serves the site from `gh-pages`.

**Tech Stack:** MkDocs Material, GitHub Actions, GitHub Pages, Python (pip for local tooling only)

**Spec:** `docs/superpowers/specs/2026-03-15-til-repo-design.md`

### Task 2: Add requirements.txt with pinned MkDocs Material

**Files:**
- Create: `requirements.txt`

- [ ] **Step 1: Install MkDocs Material**

```bash
pip install mkdocs-material
```

If you don't have Python/pip: install Python 3.x from https://python.org (pip is included).

- [ ] **Step 2: Capture the installed version**

```bash
pip show mkdocs-material | grep Version
```

Expected output (version will vary):
```
Version: 9.5.x
```

- [ ] **Step 3: Create requirements.txt with the pinned version**

Run this to generate it automatically:
```bash
pip show mkdocs-material | grep Version | awk '{print "mkdocs-material==" $2}' > requirements.txt
```

On Windows (PowerShell):
```powershell
(pip show mkdocs-material | Select-String "Version") -replace "Version: ", "mkdocs-material==" | Out-File -Encoding utf8 requirements.txt
```

Verify the file contents look like: `mkdocs-material==9.5.x`

- [ ] **Step 4: Commit**

```bash
git add requirements.txt
git commit -m "chore: add pinned mkdocs-material dependency"
```

**Files:**

```powershell
New-Item -ItemType Directory -Force docs\networking, docs\security, docs\ai-ml, docs\hardware
```

### Task 6: Verify MkDocs builds successfully

No code to write — this is a verification step.

- [ ] **Step 1: Run MkDocs build**

```bash
mkdocs build --strict
```

Expected output:
```
INFO    -  Cleaning site directory
INFO    -  Building documentation to directory: site
INFO    -  Documentation built in X.XX seconds
```

`--strict` treats warnings as errors. Fix any warnings before proceeding.

Common issues:
- `site_url` not set → update `mkdocs.yml` with your GitHub Pages URL
- Missing plugin → run `pip install -r requirements.txt`

- [ ] **Step 2: Preview the site locally**

```bash
mkdocs serve
```

Open http://127.0.0.1:8000 in your browser. You should see:
- Home page with your intro text
- Search bar in the header
- Empty navigation (no entries yet — that's fine)

Press `Ctrl+C` to stop the server.

- [ ] **Step 3: Verify site/ is not tracked**

```bash
git status
```

Expected: `site/` does not appear in untracked files. If it does, confirm `.gitignore` contains `site/`.

## Chunk 2: GitHub Actions + Deployment

### Task 7: Create GitHub Actions deploy workflow

**Files:**
- Create: `.github/workflows/deploy.yml`

- [ ] **Step 1: Create the workflows directory**

```bash
mkdir -p .github/workflows
```

- [ ] **Step 2: Create .github/workflows/deploy.yml**

```yaml
name: Deploy to GitHub Pages
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: write        # required for gh-deploy to push to gh-pages branch
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: 3.x
      - run: pip install -r requirements.txt
      - run: mkdocs gh-deploy --force
```

- [ ] **Step 3: Commit**

```bash
git add .github/workflows/deploy.yml
git commit -m "ci: add GitHub Actions deploy workflow"
```

### Task 8: Create GitHub repo and push

**Prerequisites:** GitHub CLI (`gh`) installed, or use the GitHub web UI.

- [ ] **Step 1: Create the GitHub repo**

Option A — GitHub CLI:
```bash
gh repo create til --public --description "Things I Learned" --source . --remote origin --push
```

Option B — Web UI:
1. Go to https://github.com/new
2. Name: `til`, Visibility: Public, no README/gitignore/license
3. Copy the remote URL shown

Then add the remote and push:
```bash
git remote add origin https://github.com/<your-username>/til.git
git branch -M main
git push -u origin main
```

- [ ] **Step 2: Verify the Actions workflow triggered**

Go to: `https://github.com/<your-username>/til/actions`

You should see a workflow run called "Deploy to GitHub Pages" in progress or completed. Click into it and confirm all steps pass.

If the workflow fails:
- Check the logs for the failing step
- Common cause: `requirements.txt` version doesn't exist on PyPI — check the version number

- [ ] **Step 3: Enable GitHub Pages**

> The `gh-pages` branch is created by the first workflow run — wait for it to complete successfully before doing this step.

1. Go to `https://github.com/<your-username>/til/settings/pages`
2. Under **Source**, select **Deploy from a branch**
3. Branch: `gh-pages` / `root`
4. Click **Save**

- [ ] **Step 4: Verify the live site**

Wait ~1 minute, then open:
```
https://<your-username>.github.io/til/
```

You should see your home page with the MkDocs Material theme, search bar, and tags index link.

## Reference: Adding Future Entries

For every new TIL:

1. Create `docs/<category>/<slug>.md` with the template from Task 9
2. If it's a new category, `mkdir docs/<new-category>/`
3. `git add`, `git commit -m "til: <summary>"`, `git push`
4. Site auto-updates within ~1 minute

**Tag discipline:** Reuse existing tags where possible. Keep them lowercase and hyphenated (e.g. `ai-ml`, `tcp-ip`, `gpu`). Check the `/tags/` page on your live site to see what tags already exist.

**Custom domain (future):**
1. Add `docs/CNAME` containing just your domain (e.g. `til.yourdomain.com`)
2. Update `site_url` in `mkdocs.yml` to `https://til.yourdomain.com/`
3. Configure DNS: add a CNAME record pointing to `<your-username>.github.io`
4. In GitHub Pages settings, set the custom domain field
5. Note: if you set the custom domain via the GitHub UI first, it creates a `CNAME` file in `gh-pages` — adding one in `docs/CNAME` will override it cleanly via MkDocs
